# jsonvalidate

[![CI](https://github.com/go-simd/jsonvalidate/actions/workflows/ci.yml/badge.svg)](https://github.com/go-simd/jsonvalidate/actions/workflows/ci.yml)
![coverage](https://img.shields.io/badge/coverage-100%25-brightgreen)

`Valid(data []byte) bool` â€” a drop-in replacement for `encoding/json.Valid` that
returns the **byte-identical verdict on every input** while accelerating the
hottest sub-scans of JSON validation with SIMD on **all six 64-bit Go targets**.
No cgo, no `GOEXPERIMENT`, plain `go build`.

```go
import "github.com/go-simd/jsonvalidate"

ok := jsonvalidate.Valid(data) // == encoding/json.Valid(data), always
```

Correctness is the gate, not a goal: `Valid(data) == encoding/json.Valid(data)`
is asserted on the seed table, on 60 000 deterministic pseudo-random inputs **on
every arch (including under qemu)**, and by a `go test -fuzz` differential fuzzer
on amd64 (AVX2 and SSE paths) and arm64. There is no input on which the answer
can differ â€” the SIMD kernels only ever return an index the scalar reference
would also have returned, and anything they don't accelerate runs through a
faithful scalar reimplementation of `encoding/json`'s scanner.

## Why this exists

[minio/simdjson-go](https://github.com/minio/simdjson-go) is a superb full
SIMD JSON *parser*, but it is **amd64-only**. jsonvalidate is scoped to
*validation* â€” the common "is this valid JSON?" gate â€” and trades full-parse
SIMD for **portable SIMD on every architecture Go can target with vectors**:
amd64, arm64, riscv64, loong64, ppc64le and s390x. It is, as far as we know, the
only SIMD JSON validator that covers all six.

## Scope: what is SIMD, what is scalar (honest split)

A full simdjson "stage 1" computes, for a whole block, which structural
characters fall *inside* strings. That needs a prefix-XOR over the quote bitmask,
which on amd64 is a carry-less multiply (`PCLMULQDQ`) with **no portable
equivalent** on the other five targets. A single full-SIMD grammar across six
arches is therefore not feasible. So jsonvalidate keeps the grammar â€” bracket
balancing, number/literal syntax, the exact 10 000-deep nesting limit, and every
`encoding/json` acceptance rule â€” in a **correct scalar state machine**, and uses
SIMD only for the two scans whose answer is a pure, position-independent "find
the first byte in (or not in) a small set":

- **`scanString`** â€” from just after an opening quote, find the first `"`, `\`,
  or control byte (`< 0x20`). (`encoding/json` does *not* validate UTF-8 inside
  strings, so a byte-class scan is exactly right here.)
- **`skipSpace`** â€” find the first byte that is not JSON whitespace
  (` `, `\t`, `\n`, `\r`).

Both kernels are **shuffle/compare only** â€” a per-byte nibble-LUT classification
(`c` is in the set iff `loLUT[c&0xF] & hiLUT[c>>4] != 0`) plus a first-match
extraction. No multiply, no cross-lane carry, so the same algorithm ports to all
six arches. They are pure accelerators bolted onto an always-correct scalar
validator.

| arch | ISA | classify | first match |
|---|---|---|---|
| amd64   | SSSE3 + **AVX2** | `(V)PSHUFB` Ă—2, `(V)PAND` | `(V)PMOVMSKB`+`BSF`; AVX2 (32-byte) auto-selected at runtime via `golang.org/x/sys/cpu` |
| arm64   | NEON | `VTBL` Ă—2, `VAND` | laneâ†’GPR via `VMOV V.D[i]`, `RBIT`+`CLZ` |
| riscv64 | RVV  | `VRGATHERVV` Ă—2, `VANDVV` | `VMSNE`/`VMSEQ` + `vfirst.m` |
| loong64 | LSX  | `VSHUFB` Ă—2, `VANDV` | laneâ†’GPR via `VMOVQ V.V[i]`, `CTZV` |
| ppc64le | VSX  | `VPERM` Ă—2, `VAND` | maskâ†’GPR via `MFVSRD`, `CNTTZD` |
| s390x   | vector facility (**big-endian**) | `VPERM` Ă—2, `VN` | `VFENEBS` (find-element-not-equal w/ CC), index via `VLGVB` |

VSX is baseline on POWER8+ and the vector facility is baseline on z13+, so
ppc64le and s390x (like riscv64 and loong64) need no runtime feature dispatch â€”
the SIMD path is the arch's only path, build-tagged. **s390x is the one
big-endian target:** `VL` puts the lowest memory address in lane 0 and `VFENEBS`
scans lane 0 upward, so the returned byte index is already the memory offset of
the first match â€” the classification LUTs are verified in big-endian lane order
by the position-dependent random/fuzz tests. Any 64-bit arch without a kernel
(none today) gets a correct scalar fallback.

The assembly is generated by [go-asmgen](https://github.com/go-asmgen/asmgen)
(`scan_*_gen.go`); the `.s` files are committed. To regenerate:

```sh
go get github.com/go-asmgen/asmgen@v0.5.0
GOWORK=off go run scan_amd64_gen.go   # and scan_{arm64,riscv64,loong64,ppc64le,s390x}_gen.go
go mod edit -droprequire=github.com/go-asmgen/asmgen && go mod tidy
```

## Performance

Honest numbers vs `encoding/json.Valid` (arm64, Apple silicon; representative,
not a contest):

| input | jsonvalidate | encoding/json | ratio |
|---|---|---|---|
| string-heavy (long string values) | ~710 MB/s | ~515 MB/s | **~1.4Ă— faster** |
| realistic mixed records | ~465 MB/s | ~490 MB/s | ~on par |
| number-heavy array | ~140 MB/s | ~585 MB/s | slower |
| compact-whitespace (4-space indent) | ~215 MB/s | ~490 MB/s | slower |

The SIMD scans win where they apply â€” **long string bodies**, which dominate
most real-world JSON (text, logs, API payloads). Where the document is mostly
numbers, structure, or short (< 16-byte) whitespace runs, the scalar grammar
dominates and `encoding/json`'s long-tuned scanner is faster. jsonvalidate's
value is **correctness parity across six arches with a string fast path**, not a
universal speedup. If your payloads are number-dense, `encoding/json.Valid` is
the better choice today; the scalar number scan is the obvious next thing to
optimise. `Valid` allocates nothing on any path.

## Test coverage

100% of statements, enforced in CI on every arch (native amd64/arm64; the other
four under qemu). The suite is a big table covering every accept/reject class
(literals, all number forms and malformations, strings with every escape and raw
invalid UTF-8, control chars, the exact 10 000-deep nesting boundary, structural
errors, trailing junk), a direct per-alignment test of the SIMD scan kernels
against their scalar reference, a 60 000-input differential random test, and a
`FuzzValid` differential fuzzer â€” all asserting equality with
`encoding/json.Valid`.
